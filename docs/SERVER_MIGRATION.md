# Server Migration — Dump & Restore to a Fresh Server

This guide explains how to move a running deployment (database + uploaded
media + config) to a **brand-new, freshly installed server**. It is the
companion to [`DEPLOYMENT_BACKEND.md`](./DEPLOYMENT_BACKEND.md): that document
covers a first-time install; this one covers carrying your existing data over
to a new box.

Backend is Docker-on-VM; the frontend (Vercel) is unaffected by a backend
server move — you only update the API domain it points at.

---

## 1. What Has to Move (and What Doesn't)

| Item | Location | Move it? | How |
|---|---|---|---|
| **Database** | Postgres 16 in container `hospital_postgres`; data dir `${POSTGRES_DATA_PATH:-/srv/hospital/postgres}` | ✅ Yes | `pg_dump` (logical dump) |
| **Uploaded media** | `${MEDIA_PATH:-/srv/hospital/media}` — patient/staff/verification photos, purchase-invoice documents | ✅ **Yes — this is NOT inside the database** | `tar` the directory |
| **`.env.prod`** | `hospital_backend/.env.prod` | ✅ Copy, then edit host-specific values | manual |
| **Static files** | `/srv/hospital/staticfiles` | ❌ No | regenerated by `collectstatic` |
| **TLS certificates** | `/srv/hospital/letsencrypt` | ⚪ Optional | usually easier to re-issue on the new host |

> **The single most common mistake** is migrating the database and forgetting
> the media directory. Photos and invoice documents are stored on the
> filesystem under `/srv/hospital/media`, not in Postgres. The DB only holds
> their relative paths.

---

## 2. Two Facts That Decide the Order of Operations

The procedure below is ordered deliberately because of how the backup behaves:

1. **`scripts/backup_postgres.sh` produces a _self-contained_ SQL dump**
   (`pg_dump --clean --if-exists`). Each object is dropped (`IF EXISTS`, so a
   harmless no-op on a fresh database) before being recreated, and the dump
   includes the `django_migrations` table. This means the dump can be restored
   **either into an empty database or over an existing one** — restore order
   no longer matters.

   > If you are restoring an **older** backup taken before this script change
   > (a _plain_ dump without `--clean`), it must go into an **empty** database
   > or you'll get `relation "..." already exists`. The §5 ordering
   > (postgres-up → restore → migrate) is safe for both old and new dumps, so
   > when in doubt, just follow it.

2. **Prefer restoring _before_ `migrate`.** Because the dump already carries
   `django_migrations`, a `migrate` run after the restore is a no-op (it only
   forward-applies migrations newer than the dump). The normal
   `deploy_backend.sh` runs `migrate` early; the manual ordering in §5 keeps
   restore first, which is the cleanest path for every dump format.

Other compatibility notes:

- **Postgres major version must match.** Both old and new servers use the
  `postgres:16` image, so this is satisfied. A logical `pg_dump` (used here)
  is also far more portable than copying the raw `/srv/hospital/postgres`
  data directory, which only works between identical PG builds.
- **Keep `DB_USER` / `POSTGRES_USER` = `postgres`** on the new server so the
  object ownership recorded in the dump matches an existing role.
- **Keep the same `SECRET_KEY`** unless you accept that every user must log in
  again. Stored password *hashes* survive a `SECRET_KEY` change; active
  sessions and signed tokens do not.

## 3. On the OLD Server — Create the Dumps

```bash
cd /srv/hospital/app/hospital_backend

# (a) Database — use the provided script (writes to /srv/hospital/backups/):
./scripts/backup_postgres.sh
#   …or the manual equivalent (identical command underneath):
docker compose -f docker-compose.prod.yml --env-file .env.prod exec -T postgres \
  pg_dump -U postgres -d hospital_db | gzip > ~/hospital_db_$(date +%F).sql.gz

# (b) Media (lives on the filesystem, not in the DB):
sudo tar czf ~/hospital_media_$(date +%F).tar.gz -C /srv/hospital media

# (c) Keep a copy of the env file:
cp .env.prod ~/env.prod.backup
```

If your `DB_USER` is not `postgres`, substitute it in the `pg_dump` command
(`-U <your_db_user>`).

---

## 4. Transfer the Files

Copy all three artifacts to the new server (directly, or via your laptop):

```bash
scp old-server:~/hospital_db_*.sql.gz \
    old-server:~/hospital_media_*.tar.gz \
    old-server:~/env.prod.backup \
    new-server:~/
```

---

## 5. On the NEW Server — Provision, Restore, Start

### 5.1 Install Docker and lay out persistent directories

```bash
# Docker + compose plugin (Ubuntu example)
curl -fsSL https://get.docker.com | sudo sh

# Persistent bind-mount dirs (same set bootstrap_backend_vm.sh creates)
sudo mkdir -p /srv/hospital/{postgres,media,staticfiles,backups,letsencrypt,certbot-www}

# Clone the repo
sudo mkdir -p /srv/hospital/app && sudo chown -R "$USER":"$USER" /srv/hospital/app
git clone <YOUR_REPO_URL> /srv/hospital/app
cd /srv/hospital/app/hospital_backend
chmod +x scripts/*.sh
```

### 5.2 Create `.env.prod`

```bash
cp ~/env.prod.backup .env.prod
```

Edit **only** the host-specific values, keeping DB credentials and
`SECRET_KEY` unchanged (see §2):

- `ALLOWED_HOSTS`
- `DOMAIN_NAME`
- `CORS_ALLOWED_ORIGINS`
- `CSRF_TRUSTED_ORIGINS`
- `CERTBOT_EMAIL`

### 5.3 Restore — in this exact order

```bash
cd /srv/hospital/app/hospital_backend

# 1) Bring up ONLY postgres. First init auto-creates an EMPTY hospital_db.
docker compose -f docker-compose.prod.yml --env-file .env.prod up -d postgres
docker compose -f docker-compose.prod.yml --env-file .env.prod ps   # wait for "healthy"

# 2) Restore the dump into the empty DB (script or manual).
./scripts/restore_postgres.sh ~/hospital_db_YYYY-MM-DD.sql.gz
#   manual equivalent:
# gunzip -c ~/hospital_db_YYYY-MM-DD.sql.gz | \
#   docker compose -f docker-compose.prod.yml --env-file .env.prod exec -T postgres \
#   psql -U postgres -d hospital_db

# 3) Restore media into /srv/hospital/media.
sudo tar xzf ~/hospital_media_YYYY-MM-DD.tar.gz -C /srv/hospital

# 4) Build backend, then migrate (no-op for existing tables; forward-applies
#    any migrations newer than the dump) and collect static.
docker compose -f docker-compose.prod.yml --env-file .env.prod build backend
docker compose -f docker-compose.prod.yml --env-file .env.prod run --rm backend python manage.py migrate
docker compose -f docker-compose.prod.yml --env-file .env.prod run --rm backend python manage.py collectstatic --noinput

# 5) Start the backend.
docker compose -f docker-compose.prod.yml --env-file .env.prod up -d backend
```

> Do **not** expect to reset the admin here — the superuser came over in the
> dump. `create_superuser.sh` is idempotent (get-or-create by email; never
> overwrites an existing user's password), so running the normal deploy later
> is harmless.

### 5.4 TLS + nginx

Point your DNS A-record at the new server's IP **first**, then:

```bash
./scripts/issue_tls_cert.sh
docker compose -f docker-compose.prod.yml --env-file .env.prod up -d nginx
```

## 6. Verify

```bash
# Row-count sanity check
docker compose -f docker-compose.prod.yml --env-file .env.prod exec -T postgres \
  psql -U postgres -d hospital_db -c \
  "select (select count(*) from patients_patient)            as patients,
          (select count(*) from pharmacy_dispenseinvoice)    as invoices,
          (select count(*) from staff_staff)                 as staff;"

# Media present
ls /srv/hospital/media

# App healthy
docker logs hospital_backend --tail 50
curl -fsS https://<DOMAIN_NAME>/api/v1/health/
```

Spot-check from the UI that a patient photo and a purchase-invoice document
actually load (confirms both the DB rows *and* the media files arrived).

## 7. Troubleshooting

**`relation "..." already exists` during restore**
You restored into a non-empty database (e.g. you ran the full deploy / a
`migrate` first). Reset the database and restore again:

```bash
docker compose -f docker-compose.prod.yml --env-file .env.prod exec -T postgres \
  psql -U postgres -c "DROP DATABASE hospital_db; CREATE DATABASE hospital_db;"
# then re-run §5.3 step 2 onward
```

**`role "..." does not exist`**
The new server's `DB_USER` differs from the dump's object owner. Set
`DB_USER`/`POSTGRES_USER` back to `postgres` (or the original role), recreate
the DB, and restore again.

**Photos/documents 404 in the UI but DB looks fine**
The media tarball wasn't restored, or landed in the wrong place. Confirm files
exist under `/srv/hospital/media` (this path is bind-mounted to
`/app/backend/media` inside the backend container per `docker-compose.prod.yml`).

**Users forced to re-login / signed links invalid**
Expected if `SECRET_KEY` changed between servers. Restore the original
`SECRET_KEY` in `.env.prod` to avoid this.

## 8. Self-Contained Dumps (Already Enabled)

`scripts/backup_postgres.sh` runs `pg_dump` with `--clean --if-exists`:

```bash
pg_dump -U postgres -d hospital_db --clean --if-exists | gzip > backup.sql.gz
```

`--clean --if-exists` makes the dump drop each object (only `IF EXISTS`, so it
is a no-op on a fresh database) before recreating it, so the dump can be
restored over an existing database as well as an empty one — restore order
stops mattering. New backups are therefore order-independent out of the box.

> Backups produced **before** this change are plain dumps and still require an
> empty target database (see the note in §2). To re-cut a self-contained
> backup from a running old server, just run `./scripts/backup_postgres.sh`
> again.
